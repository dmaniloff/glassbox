* TODO Fix cuSOLVER SVD convergence failures in randomized_svd
  ~5% of samples on pythia-70m (layers 4-5) trigger =torch._C._LinAlgError=
  at =svd.py:234= (=torch.linalg.svd(B, full_matrices=False)=). The cuSOLVER
  =gesdd= driver fails with error code 9 ("ill-conditioned or too many repeated
  singular values"). These are caught by the backend's =except Exception= and
  logged as =[SVD] error in layer ... at step 1=, so data is lost for those
  layer/head snapshots but the sweep continues.

  Root cause: after power iterations on a nearly rank-1 S=QK^T, the small
  B matrix (shape 9×L) becomes extremely ill-conditioned. cuSOLVER's auto-
  fallback from gesdd→gesvd also fails in some cases.

  Possible fixes:
  - Catch =LinAlgError= and retry SVD on CPU (different LAPACK, more robust)
  - Add NaN/inf guard on B before SVD call
  - Reduce power iterations (q=1 instead of q=2) for better conditioning
  - Skip SVD when condition number of B exceeds a threshold

  Observed: 26 failures in first 560 samples of pythia-70m × halueval.
  Hard to reproduce synthetically — depends on GPU memory state and cuSOLVER version.

* TODO Add span-aware AttentionTracker features (focus_score, cut_flow)
  The current attention_tracker signal computes only span-independent features
  (sigma2, sigma2_asym, commutator_norm). The AttentionTracker paper (arXiv:2411.00348)
  also defines focus_score (attention mass from data tokens to instruction span)
  and cut_flow (net directed attention flow instruction→data).
  Adding these requires:
  - A way to pass instruction/data span boundaries into the backend
    (config, per-request metadata, or marker-token detection)
  - Implementing compute_focus_score and compute_directional_cut_flow
    (reference: shade-attentiontracker/src/shade_attentiontracker/functional/attention_tracker_ops.py)
  - Both materialized and matrix-free paths
